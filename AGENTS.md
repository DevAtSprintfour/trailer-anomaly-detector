## Learned User Preferences

- Packing and anomaly analysis must be floor-level (dance vs general), not per-slot; slot numbers only classify which floor an item rode on.
- Streamlit UI exposes a single-race selectbox plus multi-select trailers (trailer options re-derived from the selected race) so analysis can be reprocessed for that exact subset.
- Prefer a simplified UI centered on two floor categories with fail/ambiguous counts and equipment drill-down, not slot-centric tabs or tables.
- When design options are clear, prefer agentic decisions over long option menus; the user often locks the recommended option quickly.
- Expanded packing viz must be one continuous trailer rectangle with dance (nose) and general (rear) sections inside — not two separate floor diagrams.
- Diagram labels must match table analysis statuses (FAIL/AMBIGUOUS), not a separate "OVERFLOW" vocabulary; pack failures are those same blame rows.
- Display packing must use the same packer (`CpPacker.pack` / `pack_best_effort`) as analysis — never invent alternate layouts that make non-fitting items look packed.
- Packing is an exact OR-Tools CP-SAT solve (ported from `Champion/anomaly-detection/plot.py`), NOT a MaxRects heuristic. Each floor is packed as its OWN rectangle against its REAL length: dance = `dance_length × width` (129×98), general = `general_length × width` (483×98). `cp_packer.pack_floor` solves one floor; `pack` / `pack_best_effort` pack both floors of a container.
- The harness gap is kept ONLY between items, never against the walls (inflate trick: bin = length+gap), so a floor-length item fits its floor exactly. IMPORTANT: do NOT add edge/corridor padding — it spuriously shrank the dance floor and flagged legitimate loads.
- **Overflow severity is per-floor** (the current blame rule, replacing leave-one-out): an item that does not fit its floor is an overflow. **dance-floor overflow → AMBIGUOUS**, **general-floor overflow → FAIL**. FAIL takes precedence when an item overflows on both across different trailers. `analyze()` implements exactly this via `pack_floor(..., best_effort=True)` and reading `unplaced` per floor; `verdict` kinds are `dance_overflow` / `general_overflow`.
- Consequence to expect: dance items are often ~130–140 in long (> the 129-in dance floor), so dance overflow (AMBIGUOUS) is common; general overflow (FAIL) is rarer. Widen a category's floor in the sidebar to clear legitimate cases.
- Container geometry is unified to one interior `width` per category (dict keys `dance_length` / `general_length` / `width`); no separate dance/general widths.
- Visualization is matplotlib in plot.py's style (`TrailerRenderer`, `st.pyplot`): dashed container outline, solid blue exclusion line + buffer corridor, tab20 boxes at solver positions, red dotted bounding-length line, `#id / name / L×W` labels. Overflow items are hatched at their floor nose — amber for dance (ambiguous), red for general (fail).
- Dims are fed to CP-SAT as integers (load-sheet dims are whole inches → exact; fractional corrections are ceiled). Solver uses `workers=1` (multi-worker hangs in the sandbox) with a memoised solve cache keyed by item/container signature; in Streamlit the packer is an `@st.cache_resource` singleton so the cache survives reruns.
- Tooling: ruff (lint+format), ty (type check), and a `.pre-commit-config.yaml` that runs both plus the test suite and `uv-lock`. ortools has no type stubs, so `unresolved-attribute` is ignored for `app/cp_packer.py` only via a `[[tool.ty.overrides]]` block.

## Learned Workspace Facts

- trailer-anomaly is a Streamlit + Pandas dimension anomaly detector (not a packing/assignment engine): load sheets are ground truth, and overflow means stored WMS dimensions are wrong.
- Analysis unit is `(race, trailer)`: one container per trailer, dance = slots 1–2 (front `dance_length`, default 129 in), general = slots 3–10 (rear `general_length`, default 483 in), shared interior `width` (default 98 in), exclusion line at `x = dance_length`.
- Packing is an exact CP-SAT solve with per-item 90° rotation and harness padding (default 2 in); blame uses leave-one-out and cross-race isolation (fixed-point iteration) into PASS / FAIL / AMBIGUOUS / UNKNOWN / RESOLVED.
- Core modules are `app/cp_packer.py` (CP-SAT engine), `app/floor_geom.py` (geometry), `app/analysis.py`, `app/packing_viz.py` (matplotlib), and `app/streamlit_app.py`; the load sheet is read live from the Champschedule (Postgres) + WMS (MySQL) databases via `app/loadsheet_source.py:fetch_loadsheet()` (credentials from `st.secrets`, falling back to the repo-root `.env`). `build_used_floors()` returns `TrailerFloor` objects (one per race+trailer, despite the name).
- Trailer categories are name-pattern based (T-Series, T-Series Top, F-Series, Other) via `app/trailer_categories.py`; each category has independently tunable container dimensions (dance length / general length / interior width) in the sidebar, all defaulting to the global values.
- Race selection is a single `st.selectbox` (exactly one race); trailer selection remains multi-select and triggers full reprocessing (`analyze()` re-run on exactly that race + trailer subset). Trailer options are re-derived from the race selection to keep them in sync.
- Verified/resolved equipment is tracked in `data/checklist.db` (SQLite, `app/checklist_store.py`) and excluded from blame candidacy in `analyze()`, forcing a `RESOLVED` status distinct from `PASS`. Dimension corrections (edited L×W) live in the same DB's `dimension_correction` table, feed `dim_overrides` into `analyze()`/`build_used_floors()`, and export as a downloadable WMS corrections CSV.
- Packing is visualized as one continuous trailer container per expanded trailer (dance chamber nose + general rear, exclusion line, visually distinguished sections) via matplotlib (`TrailerRenderer.figure` → `st.pyplot`). Placements come from the same `CpPacker` path as analysis (`display_pack`); won't-pack items are drawn hatched + dashed at their chamber nose. Per-trailer detail is lazy behind Expand/Collapse in `st.session_state["expanded_trailers"]`.
